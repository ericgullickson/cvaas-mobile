# 2. Sprint 1 Plan — CloudVision Mobile

> Reads from `1.PRD.md`. API source-of-truth: `cloudvision-apis/`. Connector .proto source-of-truth: `aristanetworks/cloudvision-python` at `cloudvision/Connector/protobuf/`. Sprint 0 closed 2026-05-11. This document is the working plan for Sprint 1 and the entry-point context for any AI agent picking up the build.

---

## 1. State at Sprint 0 close

### 1.1 Locked decisions (do not relitigate without explicit user buy-in)
- **Product**: CloudVision Mobile (working name)
- **Persona**: Onsite network technician in a wiring closet (install / MAC move / first-line access-layer troubleshooting)
- **Scope**: 3 read-only features — F1 Endpoint Locator, F2 Switch & Port Inspector, F3 Device Health Summary
- **Platform**: iOS only, Swift native (SwiftUI assumed; can revisit if UIKit serves better in practice)
- **Backend**: **None.** Hard project constraint. All API access originates from the iOS device.
- **Auth**: Service-account JWT, paste-and-store in Keychain, sent as `Authorization: Bearer <jwt>`. No OAuth/OIDC (not supported by CVaaS today).
- **Two API surfaces in the app**:
  - **REST resource APIs** (HTTPS/JSON via `/api/resources/<svc>/v1/...`) — used by F1, F3
  - **`cloudvision.Connector` gRPC** (`RouterV1.Get` over HTTP/2 with `grpc-swift`) — used by F2
- **NEAT wire codec**: MessagePack + 2 custom ExtType handlers (`PointerType`, `WildcardType`). Swift port required (~100–150 LOC).

### 1.2 Resolved risks
| Risk | Resolution |
|---|---|
| R1 telemetry path | `RouterV1.Get` gRPC over HTTP/2; NetDB path queries; native Swift client (Option A) |
| R2 auth | Service-account JWT; Bearer header |
| R7 .proto availability | Public Apache 2.0 in `cloudvision-python/cloudvision/Connector/protobuf/` |
| Live-device tenant | `cv-prod-na-northeast1-b.arista.io` (tenant `north-central-ce`), confirmed with 710P PoE switches |
| PoE field schema | Empirically confirmed via CVaaS web UI (see §1.4) |

### 1.3 Risks carried into Sprint 1
| Risk | Severity | Handling |
|---|---|---|
| R3 cellular in closets | MEDIUM | UX mitigations (last-fetched timestamps, retry); operational, no build risk |
| R4 productization not chartered | MEDIUM | Operational; no build implication beyond "TestFlight only, no App Store" |
| R5 API rate limits | MEDIUM | Client-side debounce on F1 search; surface 429 cleanly |
| R6 proto deprecations | LOW | Pin proto versions; subscribe to status.arista.io |
| R8 NetDB path stability | MEDIUM | Centralize all paths in `NetDBPaths.swift`; CI smoke test against live tenant |
| R9 NEAT codec port | LOW | ~1-day Swift implementation; port test fixtures from cloudvision-python `test/test_codec.py` |
| **NEW**: PoE & error-counter NetDB path strings | LOW (operational) | Sprint 1 discovery via Connector path enumeration against `cv-prod-na-northeast1-b`. Schema is known; only the path string is unknown. |

### 1.4 Empirically confirmed from live tenant access
- Web UI streams NetDB via a Web Worker (`cv.worker-*.js`) using WebSocket-transport gRPC (browser workaround for HTTP/2 streams). **iOS uses native gRPC over HTTP/2 directly** — same protocol, simpler transport.
- **PoE per-port fields**: `portClass`, `portState`, `priority`, `capacityWatts`, `grantedPower`, `outputPower`, `outputCurrent_mA`, `outputVoltage_V`
- **PoE device-level**: total / granted / output / available power (W)
- One 710P switch in `north-central-ce` is actively powering a Class 4 device on Ethernet7 — a useful fixture for F2 testing.

## 2. Sprint 1 goal

By end of Sprint 1, an iOS app distributable via TestFlight that — given a service-account JWT for `cv-prod-na-northeast1-b` — lets a tester:

1. Look up a MAC, IP, or hostname and see resolved switch + port + VLAN + likelihood (F1, full)
2. Pick a switch and see its health summary: model, EOS version, uptime, streaming status, active alerts, last 24h events (F3, full)
3. (Stretch) View a switch's port table with link state and at least one Connector-fetched attribute rendering correctly (F2, first-light)

**Firm targets**: F1 + F3 complete to PRD acceptance criteria. **Stretch**: F2 first-light end-to-end (one successful `RouterV1.Get`, NEAT decode of one notification, link-state rendered).

Full F2 (all 5 columns, PoE, error counters, port-detail view) is Sprint 2.

## 3. Workstreams

### WS-A: Project bootstrap *(1–2 days, blocks WS-B/C/D)*
**Goal**: Xcode project with SPM dependencies, app shell, Keychain-backed auth.

| Task | Detail |
|---|---|
| A.1 Xcode project | iOS app target. Bundle id placeholder `com.gullickson.CloudVisionMobile`. iOS 17+ deployment (bumped from 16 to use `ContentUnavailableView` and friends; iOS 17 has been out 2.5+ years, ample install base). arm64 only. |
| A.2 SPM dependencies | `grpc-swift` (apple/grpc-swift), `swift-protobuf` (apple/swift-protobuf), `MessagePack.swift` (Fluidpictures/MessagePack.swift or hirotakan/MessagePacker — pick one with ExtType support). |
| A.3 App shell | Tab bar: `Locate` (F1), `Switches` (F2/F3 entry), `Settings`. Three empty SwiftUI views. |
| A.4 Auth settings | Settings screen with two text fields: `Tenant URL` (e.g. `https://www.cv-prod-na-northeast1-b.arista.io`) and `Service Account JWT`. Store in Keychain (use `Security` framework directly or `KeychainAccess` SPM). Validate URL format. |
| A.5 Auth state | `AuthStore` observable that exposes `(tenantURL, jwt)` to the rest of the app. Tabs are gated behind auth-set. |

**Acceptance**: builds and runs in simulator. Token persisted across app restarts. Tabs show feature stubs when auth is set; show "configure in Settings" empty state otherwise.

### WS-B: F1 Endpoint Locator *(2–3 days, parallel after WS-A)*
**Goal**: F1 fully working against the live tenant.

| Task | Detail |
|---|---|
| B.1 HTTP client | `CVHTTPClient` wrapper around `URLSession`. Auto-attaches `Authorization: Bearer <jwt>`, `Accept: application/json`. Decodes errors (401, 403, 404, 429, 5xx) to typed enum. |
| B.2 Codable models | `EndpointLocationResponse`, `Device`, `Location`, `Identifier`, `Likelihood`, `Explanation`, `MacType`, `IdentifierType`, `IdentifierSource`. Match the proto exactly — see `cloudvision-apis/arista/endpointlocation.v1/endpointlocation.proto`. |
| B.3 Search view | Single text field (`Enter MAC, IP, or hostname`), submit button, scrollable results list. |
| B.4 Result row | Hostname / device-id (whichever is set), interface, VLAN, likelihood badge (color-coded), relative learned-time, explanation chips. |
| B.5 Loading / empty / error states | "No results" distinct from "Auth failed" distinct from "Service error". Empty input shows "Recent searches" (UserDefaults, 10-item cap, cleared on Settings sign-out). |
| B.6 Smoke test | Manually verify against a known MAC on `cv-prod-na-northeast1-b`. |

**Endpoint**: `GET /api/resources/endpointlocation/v1/EndpointLocation?key.searchTerm=<value>`

**Acceptance** (from PRD §5a iv):
- Sub-3s lookup time on cellular for a known MAC
- Multiple candidates rendered ordered by likelihood
- "Not found" vs. "service/auth error" visually distinct

### WS-C: F3 Device Health Summary *(2 days, parallel with WS-B)*
**Goal**: F3 fully working against the live tenant.

| Task | Detail |
|---|---|
| C.1 Device list | `GET /api/resources/inventory/v1/Device/all` (or `POST` with `partialEqFilter` for active-only). Codable `Device` model: `deviceId`, `hostname`, `fqdn`, `modelName`, `softwareVersion`, `systemMacAddress`, `bootTime`, `streamingStatus`. |
| C.2 Single device | `GET /api/resources/inventory/v1/Device?key.deviceId=<id>`. |
| C.3 Device picker | Searchable list (by hostname / IP / model). |
| C.4 Alerts | `alert.v1` filtered by `deviceId`. Codable models match proto. |
| C.5 Events | `event.v1` filtered by `deviceId` + time-range (last 24h). Cap to 20 entries. |
| C.6 Summary screen | Header: hostname, model, EOS version, uptime (computed from `bootTime`), streaming-status dot. Body: alert count → tap-through to alert list; recent events table. |

**Acceptance** (PRD §5c iv):
- <3s load on cellular for one device
- Streaming-status matches web UI for the same device
- For a known-faulty test device, ≥1 alert and ≥1 event render correctly

### WS-D: F2 first-light *(3–4 days, parallel after WS-A; stretch target)*
**Goal**: Prove the Connector architecture end-to-end. Render link state for at least one switch.

| Task | Detail |
|---|---|
| D.1 Proto codegen | Vendor `router.proto`, `notification.proto`, `ca.proto`, `sharding.proto` from `cloudvision-python/cloudvision/Connector/protobuf/`. Add `protoc-gen-swift` + `protoc-gen-grpc-swift` build plugins. Generated Swift stubs land in `Generated/Connector/`. |
| D.2 NEAT codec port | Port `cloudvision/Connector/codec/{encoder,decoder,custom_types}.py` to Swift on top of MessagePack.swift. Define Swift `Wildcard` and `Path` types matching the Python `Wildcard` and `Path` classes. ExtType codes: pull the integer codes from `custom_types.py`. Unit tests: round-trip fixture bytes captured from running the Python codec on the same inputs. Reference: `cloudvision-python/test/test_codec.py`. |
| D.3 gRPC channel | `ConnectorClient` wrapping `grpc-swift`'s `GRPCChannel`. HTTP/2 + TLS, hostname from `AuthStore.tenantURL`, JWT injected via `ClientInterceptor` adding `Authorization: Bearer <jwt>` metadata. |
| D.4 First Get | Call `RouterV1.Get` with a single query: `Query{ dataset:{type:"device", name:"WTW25120369"}, paths:[{path_elements: [NEAT.encode("DatasetInfo"), NEAT.encode("Devices")]}] }`. Confirm we receive at least one `NotificationBatch`, decode at least one `Notification.Update`'s key and value via NEAT. **Stops Sprint 1 if this fails — re-spike before continuing.** |
| D.5 Link-state path | Once D.4 works, repeat with the verified path from `cloudvision-python/examples/Connector/get_intf_status.py`: `["Sysdb", "interface", "status", "eth", "phy", "slice", "1", "intfStatus", Wildcard]`. Render the results in a stub port-table view. |
| D.6 PoE path enumeration | With Connector working, enumerate `Sysdb` to find the PoE path. Try sub-paths in order: `["Sysdb", "hardware", "poe", ...]`, `["Sysdb", "interface", "status", "eth", "phy", "slice", "1", "poeStatus", *]`, `["Sysdb", "interface", "poe", ...]`. Capture working path strings into `NetDBPaths.swift`. |
| D.7 Port-table stub | Minimal SwiftUI list bound to the link-state query results. One row per interface, link state column only. Pull-to-refresh wired to a new `Get` call. |

**Acceptance for first-light** (Sprint 1 stretch only):
- ≥1 `RouterV1.Get` call succeeds against the tenant with a real JWT
- NEAT decoder produces correct strings/ints/lists from at least one `Notification.Update`
- Port-table stub renders link state for the test 710P switch

Full F2 (5 columns, PoE, errors, port-detail screen) → Sprint 2.

## 4. Sequence and parallelism

Rough day-by-day for a solo (Eric + AI agents) build:

| Day | Workstreams active |
|---|---|
| 1–2 | WS-A (bootstrap) |
| 3–5 | WS-B (F1) ‖ WS-C (F3) ‖ WS-E.1, E.2 (plumbing) |
| 6 | WS-D.1, D.2, D.3 (Connector + NEAT setup) |
| 7 | **WS-D.4 spike** — pass/fail gate for Sprint 1 F2 stretch |
| 8–9 | WS-D.5, D.6, D.7 (if D.4 passes) **or** F1/F3 polish + Sprint 2 prep (if D.4 fails) |
| 10 | TestFlight build to Eric's phone + closet-walk smoke test |

**Critical path**: WS-A → (WS-B ‖ WS-C) → TestFlight. F2 first-light is parallel and stretch.

**Hard pause point**: WS-D.4. If `RouterV1.Get` against `cv-prod-na-northeast1-b` does not return at least one `NotificationBatch`, the F2 architecture assumption is wrong. Stop, diagnose (auth scope? TLS? proto mismatch?), fix or escalate before continuing to D.5+.
