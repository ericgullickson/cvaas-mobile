### 1.5 F.1 Discovery findings (run 2026-05-12 against `WTW25120383` on `cv-prod-us-4`)

Reproduced via `path-discovery/discover_paths.py` (Python venv at `.venv/`, cloudvision 1.30.1).

**Path-query semantics correction (carries to NetDBPaths.swift):**
The Sprint 1 plan documented Wildcards inside `path_elements` (e.g. `[..., "intfStatus", *]`). The corrected `RouterV1.Get` form to enumerate children at a path is:
- `Path.path_elements = [<segments>..., Wildcard()]`
- `Path.keys = []` (empty list, **not** `[Wildcard()]`)

Putting the wildcard in `keys` returns zero notifications even on known-good paths. This was the bug that initially made every Sprint 2 probe look empty. Reflect this in the Swift `PortStateService` extension when wiring `RouterV1.Get` calls.

**F.1.c PoE per-port — RESOLVED.** Path: `["Sysdb", "environment", "power", "status", "poePort", *]`

The PRD §5 F2 §v field schema is confirmed on this 710P. Mapping from PRD field to actual NetDB key:

| PRD field | NetDB key (per-port record) | Sample value (Ethernet14, live Class-4 PD) |
|---|---|---|
| `portClass` | `portStatus.pdClass.Name` | `"class4"` (also seen: `"classUnknown"`) |
| `portState` (Detecting/Powered/Searching) | `portStatus.poePortState.Name` | `"powered"` |
| `priority` | `portPriority.Name` | `"low"` |
| `capacityWatts` | `capacity.value` | `60.0` |
| `grantedPower` | `portStatus.grantedPower.value` | `25.5` |
| `outputPower` | `outputPower.value` | `9.30` |
| `outputCurrent` (PRD said mA; actual is **A**) | `outputCurrent.value` | `0.175` (= 175 mA) |
| `outputVoltage` | `outputVoltage.value` | `53.75` |

Additional useful per-port fields not in PRD: `temperature.value` (port-chip temp °C), `pseApprovedPower.value` (W; `-Infinity` when not approved — special-case in UI), `lldpEnabled` (bool).

Live PoE fixture: **Ethernet14** is actively powering a Class-4 PD at 9.3 W. Use this port for end-to-end testing of the PoE column.

**F.1.d Interface error counters — HARD-GATE FAILURE; column dropped from MVP.**

Probed in this order, all returned empty or directory-stub-only:
- `Sysdb/interface/counter/eth/...` — only `alias` subdirectory, no per-port data.
- `Sysdb/interface/{ethIntfCounter,strataSwexCounter,strataDebugCounter,strataFecCounter,platformCounter,cpuQueueCounter,vlanIntfCounter,aclDropCounter}` — all are writer-config/writer-status stubs that point at locations without populated data.
- `Sysdb/interface/cli/counters` — single `vlan` entry; no Ethernet rows.
- `Sysdb/hardware/strata` and `Sysdb/hardware/phy/cabletest` — stub directories with `Path` pointers to empty sub-trees.
- `Shmem/interface/status/counter/...` — `dropCounterDirPath` references this prefix, but the `Shmem` root returns zero children when queried.
- `Smash/counters/ethIntf` and `Smash/interface/counter/...` — directory pointers exist, but probing the children (with both wildcard and exact path) yields zero notifications.
- `counterDirPath` keys on `Sysdb/interface/status/eth/intf` and `.../phy/all` are **empty strings**.

Cross-check: no `inputError|outputError|inErrors|outErrors|inDiscards|outDiscards|crc|fcs|interfaceCounter` matches across the entire `cloudvision-apis/` proto surface. Interface error counters are not exposed by any Resource API either.

**Decision**: per Sprint 2 §F.1 hard gate, drop the error-counter column from MVP F2. The 5-column port-table table becomes a 4-column table: link state, PoE, LLDP neighbor, learned MAC. Port-detail screen will show "Errors: data unavailable from CVaaS — view via switch CLI `show interface counters errors`" so users aren't misled into thinking the data is zero when it's actually unreachable.

**Other NetDB roots discovered (catalogue for future work):**
Root list visible to this JWT: `Eos, NTP, Eapi, LANZ, Shark, Logs, Sysdb, Config, Smash, Kernel, ExpVars, Storage`. The PRD-referenced paths under `Smash` for MAC table (`Smash/bridging/status/smashFdbStatus`) and ARP (`Smash/arp/status`) are confirmed reachable. `ECO/deviceInfo` (PRD-referenced) returns empty on this 710P.
