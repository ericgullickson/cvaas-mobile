## 1. Overview

### a. Product / Initiative Name
**CloudVision Mobile** (working name — confirm before public artifacts).

### b. MVP Goal
Give onsite network technicians a mobile-first, read-only tool that lets them identify which switch port a given endpoint is connected to, and inspect access-layer switch and port health, from inside the wiring closet — without VPNing back to a laptop or paging the NOC.

### c. Target Release
No hard deadline; quality-driven. Internal alpha first, then a closed TestFlight pilot. Public release deferred until pilot validates value and Arista distribution path (if any) is resolved.

### d. Owner(s)

- Sponsor: Exploratory build by an Arista employee (Eric Gullickson). **Direct manager is aware; no formal sponsorship, no product charter, no committed eng/PM resources.** Path-to-production: if pilot demonstrates value, pitch internally for official productization as the CloudVision mobile companion. Until that pitch lands, treat distribution, support, and customer-facing commitments as pre-product. Intended users on success: external Arista CVaaS customers + internal Arista FAEs.

## 2. Problem Statement

### a. Current Problem
An onsite technician in a wiring closet troubleshooting or installing access-layer connectivity today must either:
- Call the NOC or a remote network engineer to "look at the switch for me," or
- Carry a laptop into the closet, VPN, and open CloudVision in a browser

Both paths are slow and create a second point of failure: the laptop is often the device being moved/restarted as part of the troubleshooting, and the NOC adds a multi-minute call queue. The tech's phone is already in their hand; CloudVision is not on it.

**Impact:** Every minute the tech is on hold or untethered from the gear in front of them is delay added to incident MTTR and physical install/MAC-move times. Affects: contractor installers, junior internal techs, FAE field engineers, and any technician dispatched without a remote engineer on call.

### b. Why Now
- CVaaS exposes the right **read** APIs today: `endpointlocation.v1`, `inventory.v1`, `alert.v1`, `event.v1`. The data is there; no platform work required.
- Cellular coverage in commercial real estate has improved to the point where a phone-tethered tool is now realistic where laptop-based was the only option five years ago.
- LLM-assisted development makes a single-developer iOS app realistic in months, lowering the bar for niche-but-valuable verticals like field-tech tools.

## 3. MVP Success Definition

### a. MVP Objectives (max 3)
- **OBJ-1**: A tech standing in a closet can identify the switch + port a given MAC or IP is connected to in **<15 seconds**, without a laptop or remote help.
- **OBJ-2**: A tech can verify a specific switch's health (alerts, streaming status, last-config push) and its port-level state (link, PoE, errors, LLDP, learned MAC) at a glance from their phone.
- **OBJ-3**: A tech can run a **cable test** (TDR) or **interface cycle** (admin shut/no-shut or PoE off/on) on a single port from the phone, with explicit service-disruption confirmation, and see the result without leaving the field. *(Promoted into MVP from Phase 2 on 2026-05-13 — see §5d, R10–R12.)*

### b. Success Metrics (outcome-based)
- **Endpoint-lookup time-to-answer**: median <15s from app open to answer rendered (vs. baseline of ~2–5 min calling NOC).
- **Pilot retention**: ≥3 of any 5 pilot techs use the app at least once per onsite session after pilot week 2.
- **Preference**: ≥80% of pilot techs report they would prefer the app over calling the NOC for "what's on this port?" questions.
- **Crash-free session rate**: >99% over the pilot window.
- **Diagnostic completion (F4)**: ≥1 successful end-to-end Cable Test or Interface Cycle run per pilot tech per pilot window, *and* zero unintended outages from accidental Run (validated via in-app telemetry on confirmation dismissals vs. confirmations).

## 4. Target Users & Context

### a. Primary Users
- **Role**: Onsite network technician — installs, MAC moves, or first-line access-layer troubleshooting. Includes customer-internal junior techs, contracted installers, and Arista FAEs visiting customer sites.
- **Environment**: Mobile-only (iPhone first). Often standing or kneeling at a rack with one hand on cables. Cellular data assumed available; degrades gracefully on flaky links. **Mostly read-only.** The narrow write surface in MVP is F4 (Cable Test + Interface Cycle) — both gated by an explicit "this will disrupt service for ~10 s" confirmation and routed through CVaaS ChangeControl for audit. No other state-modifying operations.
- **Skill profile**: Comfortable with switch-port / VLAN / MAC / IP / LLDP concepts. Not necessarily a senior engineer. Frequently a contractor or junior internal tech.

### b. Primary Use Context
- **Install / MAC move**: "Did this PC come up on the right port and VLAN?"
- **First-line troubleshooting**: "This port is dark — is the switch up? Is anything connected? Any active alerts?"
- **Equipment audit / walk-through**: "What's currently plugged into this switch?"
- **All three** happen physically in front of the rack, where pulling out a laptop is awkward or impossible.

## 5. MVP Feature Summary (4 Features)

### a. Feature 1: Endpoint Locator

#### i. Problem Addressed
Identifying which switch and port a given MAC or IP lives on today requires VPN + CloudVision web UI + multiple clicks. Slow and laptop-bound.

#### ii. User Story
As an onsite tech, I want to enter or scan a MAC/IP and see which switch and port it's connected to with a confidence indicator, so I can find the physical port without calling the NOC.

#### iii. Core MVP Capabilities
- **CAP-1.1**: Free-text search by MAC, IPv4, IPv6, or hostname — the four `IdentifierType` values supported by `endpointlocation.v1`.
- **CAP-1.2**: Result card shows: device hostname/id, interface, VLAN, MAC-learning type, likelihood/explanation, learned-time. Renders multiple candidates ordered most-to-least likely.
- **CAP-1.3**: Recent-searches history (device-local, cleared on logout).
- **CAP-1.4** *(stretch)*: Camera-based barcode/QR scan to populate the search field from an asset label. Cut for v1.0 if it slows shipping.

#### iv. Accepted When
- Given a MAC known to be on a specific lab switch, the app returns that switch + port in **<3 s** end-to-end on cellular.
- Ambiguous searches (multiple matches) display all candidates with their `Likelihood` and `Explanation` values, not just the top hit.
- "Not found" is visually distinguishable from "service/auth error."

#### v. API Mapping
- `GET /api/resources/endpointlocation/v1/EndpointLocation?key.searchTerm=<value>` (REST gateway over `endpointlocation.v1`).

### b. Feature 2: Switch & Port Inspector

#### i. Problem Addressed
After finding a relevant switch, the tech needs per-port state — link, PoE, errors, LLDP neighbor, MAC-on-port — without CLI access or a desktop browser.

#### ii. User Story
As an onsite tech, I want to open a switch and see a scrollable port table with each port's link state, PoE draw, LLDP neighbor, and learned MAC(s), so I can verify port health and identify what's physically connected. (Per-port input/output error counters were originally scoped but were dropped after Sprint 2 F.1 discovery confirmed CVaaS does not stream them through the Connector NetDB API; users must read the switch CLI `show interface counters errors` for that data.)

#### iii. Core MVP Capabilities
- **CAP-2.1**: Browse switches via hostname search or paginated inventory list (from `inventory.v1`).
- **CAP-2.2**: For a chosen switch, render a port table sorted by interface name with columns: link state (up / down / admin-down), PoE state + draw (W), LLDP neighbor (system name + remote port), learned MAC(s). (Error-counter column dropped post-Sprint-2-F.1 — see §v.)
- **CAP-2.3**: Pull-to-refresh; persistent "as-of" timestamp on the data.
- **CAP-2.4**: Tap a port to view a detail screen with expanded fields and any port-level events from the last 24 h (`event.v1`, filtered).

#### iv. Accepted When
- For a 48-port switch, the inspector loads in **<5 s** on cellular.
- Link state, PoE state, and LLDP neighbor match the device's `show interface status` / `show lldp neighbors` output.
- Port-detail screen explicitly surfaces "Errors: data unavailable from CVaaS — use switch CLI `show interface counters errors`" rather than rendering zeros or omitting the row, so users understand the gap is a CVaaS limitation, not a quiet bug.

#### v. API Mapping (Sprint 0 resolved)
- Device list: `GET /api/resources/inventory/v1/Device/all` (REST, `inventory.v1`).
- **Per-port state via `cloudvision.Connector`** — a custom gRPC path-query protocol distinct from the Resource APIs. **Architecture: native Swift gRPC client speaking Connector directly. No backend service** (hard project constraint). NetDB paths confirmed from cloudvision-python sources:
  - Link state per port: `["Sysdb", "interface", "status", "eth", "phy", "slice", "<slice>", "intfStatus", *]` → returns `interface`, `linkStatus.Name`, `active`
  - LLDP neighbor per port: `["Sysdb", "l2discovery", "lldp", "status", "local", *, "portStatus", *, "remoteSystem", *]` → returns `sysName`, remote `portId`, `ttl`
  - MAC table for a switch: `["Smash", "bridging", "status", "smashFdbStatus"]`
  - ARP table: `["Smash", "arp", "status", *]`
  - Device hardware info: `["ECO", "deviceInfo"]`
- **Outstanding (tracked as R7, R8 in §7):** obtain the `cloudvision.Connector` .proto for Swift codegen; confirm one-shot `Get` semantics on path queries are viable for mobile (vs. only `Subscribe` streaming).
- **PoE per-port schema confirmed empirically** from CVaaS web UI (`/cv/devices/poe/<id>`, verified against `cv-prod-na-northeast1-b` tenant on a 710P switch). Per-port PoE fields: `portClass` (Class 1–4), `portState` (Detecting/Powered/Searching), `priority` (Low/Med/High/Critical), `capacityWatts`, `grantedPower`, `outputPower`, `outputCurrent_mA`, `outputVoltage_V`. Device-level: total/granted/output/available power. Exact NetDB path still TBD (likely `["Sysdb", "hardware", "poe", "port", *, ...]` or `["Sysdb", "interface", "status", "eth", "phy", "slice", *, "poeStatus", *]`) — Sprint 1 discovery via Connector path enumeration. Schema is locked, path is not.
- **Interface error counters — DROPPED FROM MVP (Sprint 2 F.1.d, 2026-05-12).** Sprint 2 path enumeration against `WTW25120383` on `cv-prod-us-4.arista.io` confirmed counters are not streamed through any CVaaS API: every candidate Sysdb counter path is a writer-stub directory whose data sink is empty, the documented `counterDirPath` pointer is an empty string, `Shmem` (referenced by `dropCounterDirPath`) returns zero children, `Smash/counters/ethIntf` is an empty directory, and no `inputError|outputError` field exists in any `cloudvision-apis` proto. Visible in the CVaaS web UI on the Ethernet Interfaces page only because the web UI does not use the public API surface. Reachable for end users via switch CLI `show interface counters errors`. Re-open if a future EOS/CVaaS release publishes counters through a documented API. See `3.SPRINT-2-PLAN.md §1.5` for the full probe list.
- **Architecture note**: CVaaS web UI streams NetDB data through a Web Worker (`cv.worker-*.js`) over WebSocket-transport gRPC (browser workaround for HTTP/2 streams). The iOS app will instead use **native gRPC over HTTP/2** via `grpc-swift` — same protocol, cleaner transport. No WebSocket required.

### c. Feature 3: Device Health Summary

#### i. Problem Addressed
When a tech is dispatched to a closet for an unspecified "switch problem," they need a fast read on whether the switch itself is healthy before they start unplugging cables.

#### ii. User Story
As an onsite tech, I want a one-screen health summary for a switch — active alerts, streaming status, last config push, uptime, software version, hardware model — so I can rule the device in or out as the cause before doing port-level work.

#### iii. Core MVP Capabilities
- **CAP-3.1**: Pick a switch by hostname, MAC, or inventory list.
- **CAP-3.2**: Surface: streaming status (active/inactive), hostname, system MAC, hardware model, EOS version, uptime, last config push timestamp, active alerts count.
- **CAP-3.3**: Tap-through to active alert list (read-only) from `alert.v1`.
- **CAP-3.4**: Recent events list (last 24 h, max 20 entries) from `event.v1`.

#### iv. Accepted When
- Health summary screen loads in **<3 s** on cellular for a single device.
- Streaming-status accuracy matches CloudVision web UI for the same device within polling-window tolerance.
- For a known-faulty test device, at least one alert and one event are rendered correctly with timestamps and severity.

#### v. API Mapping
- `inventory.v1` Device resource for device info (read).
- `event.v1` Event resource for **both** active alerts (severity ≥ WARNING, `delete_time` unset) **and** recent events (last 24h). Device-scoping is applied client-side because nested-component matching via `partialEqFilter` is awkward; revisit if response volume becomes a problem.
- **Note**: `alert.v1` models the alert *configuration* surface (broadcast groups, SMTP settings, endpoint types) — not alert instances. What users call "alerts" in CVaaS are simply events with severity ≥ WARNING.

### d. Feature 4: Interface Diagnostics

#### i. Problem Addressed
When a tech finds a port that won't link up or won't power a device, they currently have to drive back to a desk to run a cable test or admin-cycle the port via the CVaaS web UI. From a closet, the only options are call-the-NOC or take a guess. The CVaaS web UI exposes two diagnostic actions on each interface (Cable Test, Interface Cycle); neither has been reachable from mobile until now.

#### ii. User Story
As an onsite tech with a port that won't link up, I want to (a) run a cable test against the suspect cable and read pin-pair fault distance from my phone, or (b) admin-cycle or PoE-cycle the port without leaving the rack, with a clear "this will take the port offline for up to 10 seconds" warning, so I don't have to drive back to a laptop or call the NOC for a routine recovery action.

#### iii. Core MVP Capabilities
- **CAP-4.1**: From Port Detail, surface an **Interface Diagnostics** entry point listing the available tests for that port: `Cable Test` and `Interface Cycle`.
- **CAP-4.2 — Cable Test**: Launch a TDR cable test. Display Date / Change Control status / Run-button state. On success, render per-pair (1–4) result table: pin pair, fault status (OK / Short / Open / Crosstalk / Unknown), fault length in meters, link speed where reported. Also render a "Neighbors" tab showing the LLDP-detected far-end device. Pre-run, fields show "Not Started."
- **CAP-4.3 — Interface Cycle**: Two selectable methods: (a) **Administratively** — `shutdown` followed by `no shutdown`; (b) **Power over Ethernet cycle** — disable PoE on the port while leaving admin state up, then restore power. After running, render the operational-state and PoE-state timeline (5-minute window straddling the cycle) so the tech can confirm the port came back up.
- **CAP-4.4 — Confirmation gate**: Every diagnostic run requires a separate, full-screen confirmation step that displays the disruption window (`~10s outage`) and the exact action being submitted. Single-tap commit is prohibited. The confirmation also notes that the action is routed through ChangeControl and recorded in the tenant audit log.
- **CAP-4.5 — Status polling**: After Run is committed, the app polls the resulting `ChangeControl` until terminal state, surfacing `Pending → Approved → Running → Completed / Failed` transitions inline. Polling can be backgrounded; tab-switching does not lose the in-flight run state.
- **CAP-4.6 — History**: Show the most recent prior run of each test on this port (date, who initiated, outcome) read from `action.v1` history if available. Empty state if never run.

#### iv. Accepted When
- Cable Test on a known-faulty patch cable returns a per-pair result table with one or more `Open` / `Short` rows and a non-zero `Length` reading, matching the CVaaS web UI's result for the same cable within 2m.
- Interface Cycle (Administratively) on a healthy linked port produces a visible `Down → Up` transition in the operational-state timeline and the port comes back to `linkUp` within 30 s of completion. The same flow on a PoE port using the PoE method drops `outputPowerWatts` to 0 and recovers it.
- Accidental-tap protection: the in-app telemetry records ≥1 cancelled confirmation per pilot tech across the pilot window (i.e. the gate actually gets used).
- If the tenant lacks the prerequisite Cable Test Python Action (see §7 R11), the F4 entry point on Port Detail shows a clear "Diagnostics not available on this tenant — see app About → Diagnostics setup" empty state rather than a broken Run button.

#### v. API Mapping
- **Action submission** via `action.v1` (`/api/resources/action/v1/...`) — submits a named Action (e.g. `cableTest`, `interfaceShutdown`, `interfacePoeReset`) against a device/interface target. The Action is server-side Python script logic installed in the tenant's CVaaS Studio, not a direct EOS RPC.
- **Change tracking** via `changecontrol.v1` (`/api/resources/changecontrol/v1/...`) — every Action runs inside a ChangeControl envelope. The app creates the ChangeControl, attaches the Action, then polls the ChangeControl status until terminal.
- **Result data** for Cable Test: pulled from the device's NetDB state via `cloudvision.Connector` after the Action completes (path TBD — Sprint 3 F4.1 discovery, analogous to Sprint 2 F.1 path-enumeration). Likely under `["Sysdb", "interface", "phy", "...", "cableDiag"]`.
- **Result data** for Interface Cycle: the existing port link-state path (already used by F2) shows the transitions in the post-cycle window; no new NetDB path needed.
- **Tenant prerequisite**: the Cable Test and Interface Cycle Actions must be pre-installed in the customer's CVaaS Studio. Tracked as R11 — discovery work in Sprint 3 confirms whether they ship with stock CVaaS or require Arista to publish/install them.

## 7. Dependencies & Risks

### a. Key Dependencies
- **CVaaS tenant with API access**: Dev tenant for F2 verification is `cv-prod-us-4.arista.io` (web UI `https://www.cv-prod-us-4.arista.io/cv/`, gRPC `www.cv-prod-us-4.arista.io:443`), test device `WTW25120383`. Supersedes the Sprint 0 baseline `cv-prod-na-northeast1-b` (tenant `north-central-ce`); the PoE field schema in §5 F2 §v was confirmed on the old tenant but should be tenant-independent EOS state. Pilot tenants TBD.
- **Auth — MVP-resolved, production-open**: MVP uses per-user service-account JWT (paste-and-store in iOS Keychain; sent as `Authorization: Bearer <jwt>`). Intentionally transitional. **Production end-state is OAuth/OIDC** ("Sign in to CloudVision" via `ASWebAuthenticationSession`), which requires CloudVision Mobile to be registered as a CVaaS OAuth client — an Arista-side admin task tied to R4 (productization). The CVaaS web UI's own login is gated to known OAuth clients; a third-party mobile app cannot use it without that registration.
- **Resource APIs in `cloudvision-apis/`** (F1, F3): `endpointlocation.v1`, `inventory.v1`, `alert.v1`, `event.v1`. All exposed over REST/JSON gateway (`/api/resources/...`).
- **Resource APIs in `cloudvision-apis/`** (F4): `action.v1` (Action submission + history), `changecontrol.v1` (ChangeControl envelope + status polling). Same REST/JSON gateway. Adds two new service surfaces to the mobile client that were not used in Sprint 1/2.
- **Tenant-side Python Actions** (F4): the Cable Test and Interface Cycle behaviors are not native EOS RPCs — they are server-side Python scripts that the customer's CVaaS Studio runs against the device. **Discovery still owed (Sprint 3 F4.1)**: whether these ship as stock CVaaS Actions or whether Arista needs to publish them in a Studio package the customer installs. If the latter, pilot tenants must install before F4 is usable on their devices.
- **`cloudvision.Connector` gRPC** (F2): native Swift client. The gRPC service is `RouterV1` (RPCs: `Get`, `Subscribe`, `GetAndSubscribe`, `Publish`, `GetDatasets`). For MVP we use `Get` (unary→server-stream — mobile-friendly: open, drain, close). All 4 .proto files are public, Apache 2.0 licensed, in `cloudvision-python/cloudvision/Connector/protobuf/` (`router.proto`, `notification.proto`, `ca.proto`, `sharding.proto`). Swift dependencies (SPM): `grpc-swift`, `swift-protobuf`, `MessagePack.swift` (for NEAT codec).
- **NEAT wire codec** (F2): NEAT is MessagePack + 2 custom ExtType codes (`PointerType` for Path, `WildcardType` for `*`). Python reference implementation is `cloudvision-python/cloudvision/Connector/codec/` (~100 LOC total). Swift port required (R9).
- **Cellular connectivity** at pilot customer sites. Assumed available; MVP does not solve poor-signal closets.
- **Apple Developer account** + TestFlight for pilot distribution. Personal Apple Developer account ($99/yr) is sufficient.

## 8. Release & Validation Plan

### a. MVP Release Strategy
- **Sprint 0 (week 1–2)**: Auth spike (R2) + telemetry path spike (R1). Go/no-go on F2 scope.

## 9. Brand & Visual System

Visual identity is aligned to Arista corporate brand. The palette below is the **complete** set of brand colors available to the app; no other hex values should be introduced for chrome, accent, or non-status surfaces. **Functional status colors (system green / orange / red) are explicitly preserved** — link-state, alert severity, and PoE state must remain readable at a glance and brand-tinting them would degrade safety-of-information.

### 9.1 Palette

| Token | Hex | Role |
|---|---|---|
| `brand.navy` | `#16315A` | Primary. Tab-bar tint, nav-bar background (when tinted), primary action color, app-icon ground. |
| `brand.steel` | `#25678D` | Secondary. NavigationLink chevrons, secondary actions, link/info icons. Replaces default system blue in non-status surfaces. |
| `brand.sky` | `#5588B7` | Tertiary / interactive emphasis. Pressed/selected state, focus rings, hero-band accents. |
| `brand.graphite` | `#58595A` | Body text emphasis, key/value labels (replaces default `.secondary` where stronger contrast is wanted). |
| `brand.slate` | `#929598` | Secondary text, captions, disabled state. |
| `brand.mist` | `#BBBDBE` | Dividers, skeleton placeholders, subtle backgrounds, capsule chip fills. |

### 9.2 Color usage rules

1. **Do not recolor status semantics.** Link state (up/down/notPresent), alert severity (info/warn/critical), PoE state (powered/searching/fault), and event icons MUST use the iOS system semantic colors (`.green`, `.orange`, `.red`, `.gray`). Field techs scanning a port list in a dim closet rely on culturally-loaded color cues; remapping these to brand blues would be actively dangerous.
2. **Brand colors carry chrome and navigation, not data.** Navy/steel/sky tint tab bars, nav bars, primary actions, and section accents. They never encode device state.
3. **Grayscale ramp `graphite → slate → mist` replaces ad-hoc `.secondary` opacity tricks.** Existing `Color.secondary.opacity(0.15)` and `Color.gray.opacity(0.2)` capsule fills become `brand.mist`.
4. **Dark mode parity is a soft requirement** (deferred to WS-G Polish). When implemented, navy stays as the primary identity color and the grayscale ramp inverts; status colors are unchanged.

### 9.3 App icon

The app icon source is `cloudvision-logo.png` (2048×2048 PNG, RGBA) at the repo root. Asset-catalog wiring lives in `ios/CloudVisionMobile/Assets.xcassets/AppIcon.appiconset/` (to be added in the polish workstream).
